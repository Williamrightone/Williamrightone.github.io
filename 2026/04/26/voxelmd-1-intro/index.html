VoxelMD | 1. Markdown 文件管理與智慧語音專案

VoxelMD 是一個 Markdown 文件管理系統，採用 Spring Boot 後端微服務以及 Vue 3 開發，提供用戶線上編輯 Markdown 文件並即時 Preview 的功能，支援照片上傳與即時預覽。

VoxelMD 同時具備 Text to Speech (TTS - Kokoro) 的服務模組，能將文件內容轉換為人聲語音播放，並支援用戶錄音與比對；針對生僻字可以建立單詞卡，反覆管理與練習。

點我看 HTML Prototype

Beta 版服務正在搭建 Infra, 以前在法商呼喊一下就有人幫你弄好, 現在要自己搭記憶體又超貴, 不過當時遇到法國教師罷工又是另外一個故事了…

關於自己造輪子

起因是今年三月底我開始整理過去的專案，這次除了重新把部分的模組模擬出來以外，我還想將這些文件寫成中英日三個語言的版本，在完成後可以順便練習一下外語。

我自己是使用 Obsidian 管理文件，寫完檔案後嘗試練習口說，但總覺得不太 Native，不熟悉的字還要額外開一個分頁做翻譯和重播；期間我也找了幾個工具與服務,但都不太滿意。

首先是 Obsidian 的 Edge TTS Plugin，回應速度很慢（甚至比我 VoxelMD 的 Local Demo 還慢）, 有時是甚至沒反應，除了 plugin 維護的問題外，應該也和 API 的網路頻寬有關。

Google 翻譯，沒毛病，只是念起文章來就沒靈魂。

typecast.ai 還真是一個不錯的產品，我使用了 Free Plan，他可以把文章段落切成多個語音段並用 TTS 服務生成語音，但可惜不支援 Markdown，他會把 code block 裡的 public static void 整段唸出來，也不支援圖片，終歸是一個專注 TTS 的產品。

基於上述，我才決定打造一個工程師專用的 Markdown + TTS 服務，一次性解決我自己的需求。

想想我碩士也是專攻 UX，出社會當分析師的時候就想這些 混蛋 忙碌的工程師怎麼就是做不出好產品是有多困難，我看我自己下去動手都比較強，結果捲袖子跳下去幹之後，發現還真的難啊…

這次規劃也發現了非常多有趣的問題，例如：

  • Markdown 即時編輯，如何使用 S3 PreSign URL（非公開影像）做照片呈現
  • 微服務框架的設計，contract、ThreadLocal 的 trade-off
  • TTS 如何 ignore Markdown 語法，不要念出 ``` 內的程式
  • Vibe Code 如何應用在中型專案內
  • 檔案如何做安全檢查
  • 英文音標要怎麼生成

以上問題會在後續文章逐一拆解，跟大家分享。

Vibe Coding

去年花了快 1 個月從設計開發到部署 Casha Pos，今年實在是沒力氣自己手搓整套，特別是有一陣子沒碰前端，依然記不得要怎麼置中一個 div。

但好在之前 Casha Pos、Rstar、Colorly 等專案都有寫下詳細的開發規格與規範，這個月果斷把 Claude Code 從 Pro 升級為 Max，進入狂點 Yes 的模式，不過在設計和 Review 的過程還是掉了不少頭髮。

專案架構

VoxelMD 後端採微服務架構，前端為 Vue 3 SPA，整體服務切分如下：

各服務職責簡述：

  • Web BFF — 前端唯一入口，負責 JWT 驗證、跨服務聚合（SAGA Orchestrator）。
  • UAA Service — OAuth2 / JWT 簽發與用戶帳號管理。
  • Article Service — Markdown 文件 CRUD、資料夾管理，以及單詞卡（生僻字）資料。
  • TTS Service — 串接 Kokoro 模型生成語音，並做 article_id + content_hash 的快取。
  • Recording Service — 用戶錄音上傳與管理。
  • Media Service — 圖片、影片上傳，含 ClamAV 掃毒與縮圖生成。

支撐基礎建設：MySQL 8（每服務獨立 DB）、Redis 7（快取 / SAGA 狀態 / JWT 黑名單）、Garage S3（自架物件儲存）、RabbitMQ（命令佇列）、Kafka（事件流）、Kubernetes + Nginx。

每個服務同時有對應的 -contract 模組，提供 DTO 與 Feign 介面給 BFF 共用，避免跨服務直接耦合內部實作。

結語

預計周三就可以把 VoxelMD Beta 版本放到架子上了, 原本只是想要做一個自己用的工具, 結果設計的時候想想好像可以做得更全面一點, 很多不錯的想法例如語音比對與評分等就安排到 Phase 2 (或是永遠不會)吧。

另外雖然說靠 Vibe Code 開發, 但是這次也是花了快三周的時間, 得益於上次做 Casha Pos 的經驗, Phase 1 的中後期速度有顯著的提升, 甚至三天可以把 Max 的 Weekly limits 用完(某方面也是 token 的用量也需要優化一下), 而由於我目前的主力電腦還是 R5 4600h + 32GB ram, 真的有一股衝動想要買 M5 pro 來跑跑看。
