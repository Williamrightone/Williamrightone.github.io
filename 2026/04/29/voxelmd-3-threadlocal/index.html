VoxelMD | 3. 從 NPE 到架構重構, ThreadLocal 跨服務設計

1. 起因

VoxelMD 的 BFF 提供一支用量彙總的 api GET /api/users/me/usage，讓前端首頁一次拿到：

  • Workspace 用量（來自 Article Service）：workspace 數、文章總數
  • Media 用量（來自 Media Service）：照片數、影片數
  • Recording 用量（來自 Recording Service，Phase 1 為 stub）：錄音數

為了避免三次 round-trip，BFF 的 GetUsageUseCase 用 CompletableFuture.supplyAsync 同時打三支下游 Feign，再把三份結果彙整回傳。

這樣每一站都在讀 ThreadLocal。看起來很優雅。

2. NullPointerException

我第一反應是：media-service 的 Controller 拿不到 UserContext??

但這很奇怪, media-service 明明有 InternalAuthFilter，他會從 X-User-Id header 還原 UserContext 再塞到 ThreadLocal。啊怎麼 Filter 失靈了?

所以問題只在這支 API 會發生。

4. ThreadLocal 與 supplyAsync 互撞

查了一下資料和問了 AI, 得到以下的結論:

4.1 ThreadLocal 的本質

ThreadLocal 是每條執行緒各有一份私有儲存空間。JVM 內部長這樣：

someThreadLocal.get() 實際上是從目前執行緒（Thread.currentThread()）的 threadLocals 查值。不同 thread 各有各的 map，彼此看不到對方。

4.2 Servlet 同步模型讓我們產生錯覺

Servlet 容器處理一個 HTTP 請求時，整條請求鏈（Filter → DispatcherServlet → Controller → Service → Feign client）都在同一條 request thread 上同步跑。於是 ThreadLocal 從頭到尾都讀得到同一份值。

Spring 生態裡大量設施都吃這個紅利：

  • SecurityContextHolder（Spring Security 身份）
  • RequestContextHolder（HttpServletRequest）
  • LocaleContextHolder（語系）
  • MDC（Logback / SLF4J 的日誌關聯欄位）
  • TransactionSynchronizationManager（當前 transaction）

這個模型的好處是 context 可以自動跟著請求走，不用每個 method 都塞 userId 參數。

4.3 CompletableFuture.supplyAsync 破壞了這個假設

兩個重載：

前者把工作丟到完全獨立的執行緒池，新 thread 的 ThreadLocalMap 是空的。任何依賴 context 自動跟著的程式碼在這裡都會破功, 例如 SecurityContextHolder、MDC、TransactionSynchronizationManager、我們自己寫的 UserContextHolder 全部都會中招。

同類的陷阱還有：@Async、裸 ExecutorService.submit()、Reactor scheduler、ForkJoinPool.invoke() 等等

5.備選方案

面對這個 bug 我想了四個方向：

方案 A：在每個 lambda 手動 set/clear

沒毛病, 只是有點太打補丁的行為.

  • 優點：改動最小，每條 supplyAsync 自己負責搬運。
  • 缺點：每個 async 點都要重複寫樣板；未來每新增一個 fan-out UseCase 都要重戰。

方案 B：抽出 ContextAware.with(ctx, supplier) 包裝器

對重複樣板做抽象。比 A 乾淨一點，但責任還是在呼叫端。

方案 C：自訂 ThreadPoolTaskExecutor + TaskDecorator

讓 executor 每次 submit 時自動把 UserContext（與 MDC）從提交 thread 搬到 worker thread。呼叫端只要指定 executor，完全不用寫任何 set/clear。

看起來是解決之道, 我直接針對這個問題提出適應性補救方案。

  • 優點：一次解決；未來所有 async 點都免費繼承；同時解 MDC 問題。
  • 缺點：多兩個類別；需要嚴守「所有 async 都走這個 executor」的紀律。

方案 D：放棄平行，改同步呼叫

三次 Feign 順序打。絕對不會踩 ThreadLocal 陷阱，但延遲變成三個的總和，違反這支 API 當初要 fan-out 的初衷。

6. 好像哪裡怪怪的

正當我和 AI 討論要選哪一個的時候, 忽然想到之前工作中使用 BFF 架構, 多數不會讓 Biz-Service 去擁有 Session, 轉念一想:

為什麼這個系統裡，身份這件事需要跨 thread 傳遞？

因為目前的 UseCase 介面是這樣：

這個 method signature 說呼叫我不需要任何輸入。但實際上它依賴 userId, 只是把 userId 藏在 ThreadLocal 裡。

這是 Clean Architecture 在講的隱式依賴（hidden dependency）：介面上看不見的耦合，讓 UseCase 在跨 thread 場景下突然失效。若我用方案 C 用一個 decorator 修好了這件事，其實沒有解決拿掉隱式依賴本身。

類比：@Transactional 和 SecurityContextHolder 也是這樣, 但它們都是框架級別的橫切關注，有文件、有固定語意、全社群預期這樣用。

因次一開始為了方便, 我設計 BFF 到每個下游 biz-service 都共用一份 UserContextHolder, 這不是框架定義的模式，是我自己複製出來的。我把框架級別的慣用語，搬到了我自己的業務邊界上。 這是值得重新評估的。

回頭看身份實際需要出現在：

  • BFF 入口：JWT 驗完後總得暫存一下供同一個請求用。ThreadLocal 在這裡是合理的。
  • BFF → 下游的網路邊界：這是跨 process，傳遞機制本來就是 HTTP header，不是 ThreadLocal。
  • 下游服務內部：userId 是業務輸入。Controller / UseCase / Service 的 method signature 本來就應該把它當成參數。

結論：

ThreadLocal 只在 BFF process 內部合理，跨網路用 header，下游應該全部顯式傳參。

7. 拔掉 biz-service 的 ThreadLocal

我決定改變前提，做一次完整重構 (又要推石頭啦!!)：

7.1 BFF 端

  • 保留 JwtAuthFilter + UserContextHolder（request 入口暫存）。

  • Port 介面 每個 method 第一個參數加 UserContext ctx：

  • UseCase 只在入口讀一次 UserContextHolder.get()，然後把 ctx 當參數傳給 Port。
    在 supplyAsync 的 lambda 裡，ctx 自然被 closure 捕獲，跨 thread 沒事。

  • Feign API 每個方法宣告所需的 @RequestHeader：

  • 刪除 UserContextFeignInterceptor：不再有「讀 ThreadLocal 塞 header」的隱式行為；
    每個 Feign API 自己宣告需要哪些 headers。

  • UserContextTaskDecorator → MdcTaskDecorator：decorator 只剩 MDC 傳遞（因為
    log correlation 無法走參數）；身份傳遞交還給 method signature。

7.2 下游三個 biz-service 端

  • 刪除 每個 service 的 InternalAuthFilter。

  • Controller 用 @RequestHeader("X-User-Id") Long userId 顯式接參數：

  • UseCase / Service 全部把 userId（或 Plan）做成顯式 method 參數，不再讀
    UserContextHolder。

7.3 原本 async bug 從根本上消失

對比一下兩個方案：

項目 方案 C（decorator） 最終方案（顯式傳參）
bug 有沒有可能再發生 要記得用 decorated executor，忘了就中 不可能——userId 就是 method 參數
新 UseCase 成本 免費繼承 signature 多一個參數
下游服務 ThreadLocal 保留 完全拿掉
隱式依賴 仍透過 ThreadLocal 零（所有依賴在簽名上）
測試成本 需要 mock ThreadLocal 直接傳參，Mockito 直觀
未來加 tracing / MDC decorator 共享 decorator 只管 MDC，簡化責任
HTTP 契約 由 interceptor 隱式塞 由 Feign API 明確宣告

7.4 over-engineering

一開始想用方案 C 達到對症下藥, 但實際上午可以讓這個病症完全不會出現, 兩者看似都可以工作，但：

  1. 規則數量不同：方案 C 需要一條規則, 所有 async 必須用 decorated executor；而最終方案不需要這條規則。
  2. 可驗證性不同：方案 C 漏寫的後果是正式環境 NPE；最終方案漏寫的後果是編譯失敗。
  3. 隱式依賴的擴散：方案 C 的前提是全 codebase 都同意 ThreadLocal 為身份載體。最終方案把這個前提縮小到 BFF 入口。

當一個規則是不要犯錯時，你終究會犯錯。但當一個規則是錯了就不能編譯時, 就能提前預知

之前設計一些架構的時候, 我們會在父類別方法內直接拋錯誤, 強制子類別去 Override, 也是類似的概念.

7.5 最後的 UseCase 長這樣

注意有兩件事都做到了：

  • 用了有 decorator 的 executor（usageExecutor），所以 MDC 能跨 thread。
  • 身份是 ctx，不是 ThreadLocal, lambda 直接閉包進去。

結語

技術與架構教訓

  • 凡是 ThreadLocal 撐住的 context，一旦跨 thread 就會遺失。 @Transactional、SecurityContextHolder、MDC、RequestContextHolder 全部一樣。看到 supplyAsync沒帶 Executor，要立刻警覺。
  • CompletableFuture.supplyAsync(Supplier) 是 trap。 不僅 context 會掉，還會用 CPU-bound 設計的 commonPool 跑 blocking I/O。
  • MDC 跨 thread 需要 TaskDecorator；身份跨 thread 可以（而且最好）用 method 參數。 兩件事責任不同。
  • Bug 本身常常只是徵兆。真正的問題是設計允許這個徵兆存在。 修徵兆是短線思維，質疑前提是長線思維。
  • **ThreadLocal 屬於 framework 級別的橫切關注，不應該成為你家業務邊界的傳遞機制。**跨網路邊界請用網路協定（HTTP header），跨語言邊界請用明確契約，跨函式呼叫請用參數。
  • 判斷是否該擁抱隱式：問問「這條規則如果違反，會不會自動失敗？」 會 → 隱式 OK；不會 → 做成顯式。

在這個過程中我差點就停在方案 C, 然後趕下一個 Feature 了, 其實可以想一下為什麼這個 bug 有存在的可能性.

然後在 ThreadLocal 的跨服務使用上還是栽了跟頭, 原本在單體架構下 ThreadLocal 就很難出現這樣的問題, 果然微服務還是有很多的經驗要在實作的時候才會發現。

有趣的是在文章發布前我跟 M 老弟 (跳舞認識的工程師) 討論這個話題的時候, 他問說如果未來有新的橫切關注點加入 (例如特定的 AOP 稽核紀錄需要更多 User 資訊)，我會傾向於繼續擴充 UserContext 物件，還是會考慮重新引入某種受控的 TaskDecorator?

而我說會考慮將它封裝成一個特定的 Request Context , 新的內容就不擴增在 header 內, 而是多建立一個封裝物件.

M 老弟說他要當我上線的第一個用戶, 並分享了一段破碎的名言, 大意是簡單優於複雜, 我回去查了一下, 原來是 Python 社群的名言 (M 老弟你不是寫 Ruby on Rails 的嗎?):

“Explicit is better than implicit. Simple is better than complex.” —— The Zen of Python

最後是內心的一些小感嘆, 畢竟人在設計架構上都會犯錯, 或是沒有用到最佳的方法, 而且架構改動要敢做，但要有測試和規範兜底.

這一次的改動有 30+ signature, 依靠整個系統一開始就有 Clean Architecture 分層, 加上有 AI 可以協助更新, 其實修改起來非常快, 回想起之前工作的時候看到那些改不動的設計, 除去不熟悉業務邏輯外, 真正可怕的是程式的耦合, 因此經過這次修改後, 我也更堅信自己設計與遵守這些準則是對的。
