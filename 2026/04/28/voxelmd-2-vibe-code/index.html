VoxelMD | 2. 關於 Vibe Code 開發心得與架構規範

去年在撰寫 Casha Pos 的時候, 大多數的 AI 開發還是依靠對話窗以及少量的 Terminal 請 Agent 協助我更新小部分的程式, 或是給他範例, 讓他對應的重做出一套流程.

結論就是 Context 開始變得超大, 常發生 web 有在跑, 但是要等非常久才會有回應的情況, 變成要適時請他彙整出目前的結論與做法, 複製後貼到另外一個對話內繼續 (其實就是手動 /compact 吧).

但好在專案都有先花時間撰寫出規範, 例如專案目錄結構, DDD + Clean Architecture, naming-rules 等等, 每次手動 compact 的過程就還算順利, 但是中途還是花不少的時間在手動複製貼上, 最可怕的是到中期開發才現一些問題, 整體要調整和改的成本實在是很高。

對此, 我今年改變策略, 在做 VoxelMD 前大致上研究了一下 Claude Vibe Code 的作法, 期間也看了一下 Claude How To, 內容是真的多, 但其實有不少的工具是暫時用不到的, 可以等需求更明確後逐一 Pick Up 起來.

目錄配置

先展示一個整個 Repository 的目錄:

總體而言, 在 Repository 下共有三個 CLAUDE.md 檔案, 分別為前後端專案, 和 Root 專案使用, Root CLAUDE.md:

  • CLAUDE.md: 專案總入口、命名慣例、架構摘要、規則檔索引, 每次 session 自動使用
  • backend/CLAUDE.md: 後端 Java/Maven 細節, 進到 backend 工作時使用
  • frontend/CLAUDE.md: 前端 Vue/Vite 指令, 進到 frontend 工作時使用

根 CLAUDE.md 包含:

  • 協作偏好 — 對話用繁中、文件/程式碼註解用英文、dish.md 用途
  • 專案概述 — VoxelMD 是英文口說練習平台
  • 指令 — frontend 的 npm scripts、backend Maven 狀態
  • 文件導引 — 指向 doc/rules/* 與 API spec 慣例
    • 架構 — Service 拓撲、Maven 模組、Clean Architecture 分層、DTO 命名表等
    • 資料庫規則 — Snowflake ID、Flyway、MySQL 8 charset 文件的指向
    • 例外處理 — 5 位錯誤碼、ModelType 前綴 (US/AR/TT/RC/MD/BF)、錯誤級別
    • Key Design Decisions — SAGA、TTS cache、JWT blacklist

我刻意把 CLAUDE.md 控制在 100 多行, 它是每次 session 自動載入的稅。詳細規則拆到 doc/rules,讓 AI 按需要去讀、不要預設全載

doc/rules/ — 規則模組化

總計有 16 份MD, ~7,162 行; 按需載入 但搜尋關鍵字命中時 Claude 會整份讀, 這一部分也是最有優化空間的地方.

  • api-response.md: 統一 API 回應格式
  • auth-context.md: JWT、UserContext、顯式 header 契約
  • clean-architecture.md: 分層職責、命名、依賴方向 (我個人認為最重要)
  • database.md: Snowflake ID、Flyway、MySQL
  • events.md: Kafka events
  • exception.md: 例外階層、錯誤碼
  • frontend-architecture.md: 前端架構 (單檔最大)
  • media-upload.md: 照片/影片上傳、ClamAV、S3、配額
  • orphan-cleanup.md: 孤兒資料清理
  • recording.md: 錄音模組
  • redis.md: Redis 用途
  • testing.md: 測試規範
  • tts.md: TTS 模組
  • vocabulary.md: 字彙模組
  • workspace-content.md: Workspace 內容

doc/apis/ — Spec-per-endpoint

doc/apis/README.md 是主索引, 按服務分節列出:Feature / BFF Endpoint / Internal Endpoint / Spec 連結。

Spec 雙層慣例: 因為我是走 BFF 的 Facade Pattern, 所以一個 feature 通常有兩份 spec:web-bff/X.md(公開層)+ /X.md(內部層), BFF 透過 OpenFeign 呼叫對應 service。CLAUDE.md 明確要求「新增/修改 spec 時必同步檢查另一層」。

doc/db/ — DB schema 文件

對應 flyway<DB>\ 的 migration, 更新時兩者同步, 可以順便生成 Er-Model, 之前當 SA 的時候很愛看。

Vibe coding 流程的核心約定

從 CLAUDE.md + 規則檔可萃取出以下共識與流程:

  1. 語言分流 — 對話繁中 / 文件與程式碼英文
  2. dish.md = 暫存桶 — user 把長篇參考文件丟在這, Claude 看到就讀
  3. Clean Architecture 強制分層 — Controller → UseCase → Service → Repository/Port (Adapter)
  4. DTO 後綴語意 — Rq / Rs / Model / Vo / Dto / Entity(每個都有固定意義)
  5. 錯誤碼可推算 — {ModelType}{ErrorCode} 例如 US00001
  6. Spec 必同步 — 改 BFF 必檢查 service spec、改 service 必檢查 BFF spec、最後更新 doc/apis/README.md
  7. Snowflake ID + Flyway 版本格式 major.minor.patch + 檔名前綴 create_/alter_/insert_

再有這些資訊後, 我通常會先在 dish.md 內寫出我對這個 Feature 的 User Journey Map 和 Usecase 流程, 部分的細節, 甚至是 Wireframe, 先請他產出 Prototype, 並說明後端實作的流程, 大概會寫那些程式, 還有一些開發的後端重點。

這也是讓我很驚豔的一點, Claude 基本上已經是一個優質的前端工程師了(當然他還是得遵守我的前端架構規則)

在 Prototype 上操作後, 通常會來回核對開發的細節, 全部確認後, 就可以請 AI Agent 依照規範產出 Spec, 檢驗 Spec 無誤後, 再請 AI 實作.

不過最終還是要我親自啟動服務與測試, 之前看我的學長用 Openclaw 讓整個開發與測試全部自動化, 不過很燒 Token 就是了, 我目前還是偏好自己測試, 也比較安心, 同時對於測試環境的 flyway 也要自己執行 migrate, 也比較符合我之前在金融業開發需要設定中斷點的規範。

結論

目前的這一套架構讓我大概花兩周的時間就開發完了第一階段的核心功能, 等待 AI 撰寫的過程還能順便看書 (Youtube), 但後面 DevOps 的流程我依然是使用 private Github Runner, 並且自行搭建 SonarQube, 有各種 Code Smell 則整理後再給 AI 修正, 或許可以在專案內設置 Skill 來修正, 不過專案若需要通過 SAST/DAST, 這樣也要研究綁定到 skill 內是否能夠留下相關紀錄了(因為 SAST 報告通常需要可稽核的執行紀錄,skill 是 client-side 行為)。

以上大概是這次 Vibe Code 開發的流程, 主要是準備好專案的架構與規範, 產出的程式其實就和我自己寫的高度相似了, 這幾天應該會再詳讀一次 Claude How To, 來看看如何引入需要的工具, 以及怎樣可以優化 Token 的消耗。
