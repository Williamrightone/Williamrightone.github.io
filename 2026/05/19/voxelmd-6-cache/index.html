VoxelMD | 6. Markdown/Canvas Autosave 的 Redis 寫入緩衝設計

作為一個在台灣金融業打滾的軟體工程師, 可能是曾經歷過開發大型中台環繞 Mainframe 主機, 或是一顆巨大的 Database 的緣故, 導致我在面對需要頻繁寫入資料的情境都會特別敏感.

剛好 VoxelMD 就是一個十分經典的案例, 使用者在編輯器裡打一段文字、或在 Canvas 裡拖一個圖形(Canvas 通常會被轉譯成 json 儲存), 每一次 keystroke / pointer move 都產生一個新的文件狀態, 如果每一次狀態變更都直接寫 DB，對一個活躍使用者來說，每分鐘可能產生數十到上百次 DB write, 這既浪費資源，也讓 DB 成為延遲瓶頸.

可是如果完全不存，使用者關分頁、斷網、甚至瀏覽器 crash 時, 就會丟失所有未存的編輯. 此時必然要設計一個機制, 要能保障用戶編輯的資料不丟失, 又能不對 DB 的效能造成影響, 此時快取的技術便是一個常見的解決方法.

架構設計

開始前先思考一下技術選型:

  • WebSocket vs Redis Cache: 目前是單人編輯場景。WebSocket 的價值在於多人即時同步，但它帶來的複雜度（連線管理、斷線重連、CRDT / OT 衝突解決）在現階段不值得.
  • Spring @Cacheable: @Cacheable 是 read-through cache。我們的場景是 write buffer — 先寫 cache、再（延遲）寫 DB。語意不同。直接用 StringRedisTemplate 更明確、更可控.

VoxelMD 整套機制由三層組成，前端兩層、後端一層 (ASCII 懶的調整…)：

簡單來說, 當用戶在前端編輯時, 網頁會及時與定時發送儲存的請求, 關閉時也會強制儲存; 而後端則會用 Redis 寫入緩衝, 減緩 DB 壓力.

Article 和 Canvas 共用同一套 pattern，差異僅在並發控制策略。這裡先講共同的骨架，後面再講各自的特化設計。

前端：兩層計時器

Layer 1: Debounce（1.5 秒）

使用者每次編輯觸發 schedule()。它做兩件事：

  1. 重置 debounce timer — 如果 1.5 秒內沒有新的編輯，就發一次 flush=false 的 PATCH。
  2. 啟動 periodic timer（僅第一次）— 每 30 秒強制發一次 flush=true。

debounce timer 的設計目的是當使用者停筆 1.5 秒時，很可能只是在想下一句話, 接著馬上又會繼續打。所以這時只需要把當前狀態暫存到 Redis 即可，不值得為了一個可能馬上被覆蓋的版本, 額外花一次 DB write。

這裡也分享一下, 早期在做 UX 研究的時候, 針對 1.5 秒這種用戶的操作參數, 是需要非常嚴謹的 usability test 的, 當時的師傅 A 曾告誡我 UX 是 Scrum 的最前線, 每一個結論與抉擇都會影響著後面的 Sprint, 因此每一個設計與發現都至關重要, 後來偶拾會接觸到 PM 隨便聽 User 需求就出設計的場景, 不免心生感嘆(不知道師傅現在過得如何).

Layer 2 — Periodic flush（30 秒）

Debounce 的設計只保證停筆就暫存，但如果使用者連續不停地打字，debounce timer 會被不斷重置, 直到他終止為止。因此 Periodic timer 是安全網, 不論使用者有沒有停，每 30 秒至少落地一次。

換個角度來看，這兩層各自對應一個分散式系統的標準概念：

  • Layer 1 是 debounce（流量抑制） — 在使用者連續輸入時，抑制前端對後端的請求尖峰，避免把高頻 keystroke 直接打到 DB。
  • Layer 2 是 periodic flush（RPO 上限保證） — 它的本質是把這套機制的 RPO（復原點目標）硬性鎖在 30 秒：即使在永不停筆這種讓 debounce 永遠不觸發的極端輸入下，系統的最大資料遺失窗口也絕不超過 30 秒。

這兩層疊起來的效果：

使用者行為 觸發 flush 後端行為
停筆 1.5 秒 debounce false Redis buffer only
持續編輯超過 30 秒 periodic true Redis → DB
離開頁面（in-app nav） flush() true Redis → DB
關閉分頁 / 重整 flushBeforeUnload() true Redis → DB（keepalive）

離場保護：flush() vs flushBeforeUnload()

離場有兩種情境，需要不同的處理：

In-app navigation（Vue Router 換頁）— onBeforeRouteLeave 觸發，此時 JS runtime 還活著，可以 await flush() 等 axios 完成。

Browser-level leave（關分頁、重整、外部連結）— beforeunload 事件觸發，瀏覽器給你的時間窗極短，任何 pending 的 XHR / axios 請求都會被終止。這裡必須用原生 fetch + keepalive: true：

keepalive: true 告訴瀏覽器即使頁面已經被銷毀，也要把這個 request 送完。因為它有 64KB 的 body 上限限制（Chrome W3C），但我們的 payload 在 media 外部化之後通常遠低於此。這是 fire-and-forget, 沒有 response handling，因為頁面已經不在了。

為什麼不用 navigator.sendBeacon()？因為 Beacon 只支援 POST，而我們的 autosave endpoint 是 PATCH（canvas）/ PUT（article），且 Beacon 無法設定自訂 header（我們需要 Authorization）。

後端：Redis 寫入緩衝

兩階段寫入

後端則是收到 autosave 請求後，根據 flush 參數決定走哪條路：

這個設計的核心洞察是大多數 debounce 觸發的版本，會在下一次 flush 之前被覆蓋。 假設使用者在 30 秒內停筆 10 次（觸發 10 次 debounce），這 10 次只有最後一次的版本會被 flush 寫進 DB。前 9 次的 Redis SET 雖然各花了一次網路 round-trip，但比起 9 次 DB write（含 transaction、WAL、index update），成本低了一個數量級。

這裡也是 Redis 解除 DB 壓力的核心設計.

Redis key 與 TTL

Domain Key pattern TTL
Article autosave:content:{articleId} 60s
Canvas autosave:canvas:{canvasId} 60s

我設計 TTL 60 秒，他是 periodic flush 間隔（30s）的兩倍。這確保:

  • 正常情況下，buffer 會被 periodic flush 主動清除（30s 觸發、flush 成功後 DEL）。
  • 如果 flush 失敗（網路抖動、後端重啟），buffer 最多在 60s 後自動過期，不會殘留成 stale data。
  • 如果使用者關分頁後再也沒回來，orphan buffer 60 秒後自動消失。

讀取路徑 buffer 優先

另外, 當讀取文章或 canvas 內容時，後端先檢查 Redis buffer：

這裡有一個關鍵, RS 回傳的是 buffer 的內容 + DB 的 hash, 因為:

  • 內容用 buffer 的：確保使用者看到自己最近的編輯，即使他還沒落地到 DB。
  • Hash 用 DB 的：並發控制必須基於持久狀態。如果用 buffer 算的 hash，兩個 tab 可能各自看到不同的 buffer hash 而誤以為自己是最新的。

這個設計可以讓讀取路徑成為一種 read-repair：使用者重整頁面後看到的內容是最新的（含 buffer），而 hash 仍然正確反映 DB 的狀態。

Hash — 變更偵測，不是安全

兩個 domain 都使用 MD5 hex（32 char）作為 content hash：

這個 hash 的用途主要是偵測內容有沒有變，所以不需要密碼學安全的程度。MD5 在同一份文件的前後版本碰撞這個 scale 上，碰撞機率可以忽略。它的優點是快、結果短（32 char 存欄位剛好）、而且 Spring API 內建。

Hash 在這套機制裡扮演兩個角色：

  1. Idempotent skip — flush 時如果 hash 沒變，跳過 DB write。這避免了使用者打了一個字又刪掉這種 no-op 變更觸發不必要的寫入。
  2. Optimistic concurrency token（僅 canvas）— client 帶上次看到的 hash，後端比對不符就回 409。

Article vs Canvas：共同骨架，不同取捨

Article 和 Canvas 兩者用了同一套 debounce + periodic + Redis buffer 骨架，但在三個地方做了不同的決定：

1. 樂觀並發控制

Article Canvas
Client 送 hash 否 是（expectedSceneHash）
後端 conflict check 無 有 — 不符回 409 CV00002
設計理由 文章為單人編輯場景，後寫覆蓋前寫是可接受的 Canvas 有多人同時開啟的可能，靜默覆蓋會丟圖

Article 的 hash 純粹用於 idempotent skip。Canvas 額外把它當作 concurrency token — 這是因為 Canvas 的 scene_json 包含圖形元素的位置資訊，靜默覆蓋的代價（使用者畫的東西不見了）比文章更高。

2. Payload 上限

Article Canvas
後端檢查 無明確上限 512 KB（MAX_SCENE_JSON_BYTES）
超過時 靠 nginx client_max_body_size 擋 400 CV00003 SCENE_JSON_TOO_LARGE

Canvas 有明確的 512KB 上限，是因為 Excalidraw 的 scene_json 在 media 外部化之前，有可能因為內嵌 base64 圖片而暴漲（我原本要寫一篇文章來講這個雷點, 但我自己都覺得有一點 hardcore, 就算了）。Media 外部化後(我把 base64 改為我自己的 S3 pre-sign URL)，scene_json 裡只剩 media://{id} 指標，512KB 足以容納非常複雜的場景。

3. HTTP method 與 URL

Article Canvas
Method PUT PATCH
URL /api/v1/workspaces/{wsId}/articles/{articleId}/content?flush= /api/v1/canvases/{canvasId}/scene?flush=

Article 用 PUT 因為它是整體替換 content 欄位；Canvas 用 PATCH 因為它只更新 scene_json（canvas 還有 name、displayOrder 等其他欄位走另一個 PATCH endpoint）。

DB commit 失敗 — Redis DEL 無法參與 transaction

這是整套機制裡最微妙的一個邊界。autosave() 是 @Transactional，flush 路徑的順序是：

關鍵在於：DB 的 commit 發生在方法 return 之後，而 Redis DEL 在方法內、commit 之前就執行了。Redis 操作是非交易性的副作用，無法被 @Transactional 涵蓋、也不會隨 DB rollback 而回滾。

先釐清一個不會發生的誤解：如果 canvasRepository.save() 本身拋例外，DEL 那行根本到不了，buffer 留著、DB 一起 rollback. 這個情境是安全的。DEL 是順序語句、不在 finally。

真正的殘留風險是另一個時機點：save() 沒拋例外，DEL 執行了，方法 return，接著 transaction 在 commit 階段失敗（連線在 commit 時斷、約束在 flush 時才觸發）。此刻 buffer 已被刪、DB 也 rollback，後端兩邊都沒有這次編輯。

不過我認為實務上幾乎不會真的丟資料, 因為前端是 source of truth：commit 失敗 → 後端回 5xx → 前端不會呼叫 markSaved() → dirty 維持 true → 下一次 30s periodic tick 用記憶體裡的 scene 重送、重建 buffer。要真的遺失，必須commit 失敗 且 使用者在下次重試前關掉分頁 且 關閉時的 keepalive 也失敗三者同時成立, 這個重疊的機率低到我認為去防護反而成本過大.

Redis 掛了

後端的 StringRedisTemplate 操作會拋例外。flush=false 路徑直接失敗（Redis 是唯一目標）。flush=true 路徑也會失敗，因為 flush 前的 SET 失敗了。前端 catch 後靜默重試 — 下一次 debounce / periodic tick 再試。最壞情況：Redis 長時間不可用，所有 autosave 都失敗，使用者看到的永遠是 “Saving…”。

關於 Redis 故障降級的架構抉擇, 當前我刻意選擇不做降級。若 Redis 宕機就把 flush=false 的高頻流量 fallback 成直寫 DB，等於在 Redis 不可用的同時，把原本被緩衝吸收掉的尖峰全量灌進關聯式資料庫, 換來的是下游 DB 的連鎖崩潰（雪崩），變成我用一個更嚴重的故障取代一個較輕的故障。

另外 Redis 的高可用應該由基礎設施層（Redis Sentinel / Cluster）保障，而不是在業務應用層為一個**非核心路徑（autosave）**硬塞降級邏輯。這符合 Fail-Fast 原則, 與其讓系統帶著破損的不變式繼續跑，不如讓非核心路徑乾脆失敗、由前端 retry 兜底，把問題收斂在它該被解決的那一層。

同時在大量 autosave 都失敗的情境下, 我們的 tracing 機制也可以發出效果, 發送通知給管理員去修復 Infra 相關的問題.

讀取路徑不防快取擊穿（cache breakdown）

只要看到 getAutosaveSceneJson(canvasId).orElse(persisted) 這段 buffer-miss-then-DB 的讀取邏輯，熟悉 cache 的人應該會立刻反問, 若 buffer 過期的瞬間若有大流量同時湧入，會不會擊穿到 DB？

這裡其實回歸到使用場景:

  1. 經典 cache breakdown 之所以危險，是因為 miss 之後要做一次昂貴的重算（複雜查詢、聚合、外部呼叫），大量請求同時撞上重算才會壓垮後端。但這裡的 orElse(persisted) 只是一次主鍵索引查詢, 沒有重算，fallback 成本極低。
  2. 即使構成，這個場景也不需要防禦。 因為 autosave 的讀取路徑只有**編輯者本人（或極少數協作者）**會頻繁觸發，它不是面向大眾、會被導流放大的 read-heavy fan-out。不同於那種熱門 key 過期瞬間數千 QPS 同時 miss 的場景，所以我覺得不需要 single-flight 或是分散式鎖等機制。

所以常說, 設計沒有卍解, 只有最符合當下需求的設計, 依場景做恰當取捨才是程式設計師該有的素養.

結語

Redis 過去最長的使用方法都是, 我從 DB 內拿出來重複使用, 減少 DB 的讀取壓力, 這次是用 Redis 當一個 Buffer Pool, 減緩 DB 儲存的壓力(當然還有 Buffer 優先讀取), 個人覺得解決這個需求滿帶感的, 就和大家分享.

其中因為一人開發(還有我的 Claude 好夥伴, 前端開發變得極度簡單, 我在前端的方面都快變成 lgtm 超人了), 反而讓我更有機會思考當前的設計到底合不合需求, 以及這些參數要怎麼制定, 忽然有一種反璞歸真的感覺.
