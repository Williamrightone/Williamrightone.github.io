VoxelMD | 4. 上線 TTS 優化案例與心得

在四月中的靈光一閃, 讓我開始了 VoxelMD 的 Vibe Coding 之路, 終於在今天凌晨搞定了 ArgoCD 和 Helm Chart(真的是有點複雜), 並利用 Cloudflare tunnel 把服務導入到我自行架設的 K8s 內了.

上線第一單, 音檔生成過久

在 Beta 版上線後, 由於沒有 UAT 環境, 我開了測試帳號給 P 君(也是我跳舞認識的朋友), 結果他只回我一句話:

卡了

Damn 跟我之前的 QA 還有分析師一樣惜字如金, 忽然有一點血壓升高的感覺, 登入進去看了一下, 這個 Bastard他直接貼了一段 Serena 的 README 到裡面生成語音, 由於我設計 Kokoro Service 走非同步的流程, 前端會每格兩秒 polling 一次, 我自己測試了一下。

好嘛, 202 個 requests, 生成了一個 12 分 27 秒的音檔, 換算下來跑了大概七分鐘, UX 零分, 上線不到一分鐘直接被開單, 笑死。

問題分析

用戶行為:

  • 用戶在 Editor 完成 Markdown 內容, 點選 Regenerate 按鈕。
  • 前端對 GET /api/v1/workspaces/{wsId}/articles/{articleId}/tts?voice=af_bella 持續 polling。
  • 每次 polling 收到:
  • 直到 Kokoro 完成合成、tts-service 把音檔上傳 S3 並 setReady,才回傳 status: "ready" 與 audioUrl。

實際的 API Flow:

在盤點流程後, 就可以進行進一步的分析:

硬體拓樸:

角色 機器 規格 用途
Worker vox-worker-3 (VMware VM) 6 vCPU / 52 GB 中間件,含 Kokoro

關鍵限制:

  • worker-3 是 VMware Workstation VM,不支援 GPU passthrough。
  • Edge host 上用的 GPU 是 GT 1030 即使 passthrough 可行,2GB VRAM、無 Tensor Core, 對 Kokoro 提升僅 1.5~2x,工程成本不划算。
  • 整個架構在可預見的將來都是 CPU 推理,優化只能往 CPU 路線走。

kokoro container:

沒有設定 OMP_NUM_THREADS / ONNX_NUM_THREADS 等 thread 環境變數.

根因分析:

OMP_NUM_THREADS / ONNX_NUM_THREADS 預設行為:

  • 多數 OpenMP / ONNX runtime 在沒設定時,會偵測容器看到的 CPU 數,但在 cgroup 限制下偵測往往失準,常常退化成 單緒。
  • 即使容器看到 4 核(limits),沒明確設定環境變數,實務上 Kokoro 常常只用 1 緒。

Measure first, optimize second

由於 Beta 版本還沒有引入 Prometheus + Grafana + Jaeger, 預計先針對瓶頸方法補上 結構化計時 log + Micrometer Timer, 紀錄一下確切的前後時間; 否則改完無法證明改善幅度,也分不清是 Kokoro 變快、還是 article-service / S3 / 其他 stage 改善.

Optimization Plan

Tier 1 具體做兩件事:

(a) Stage 計時結構化 log:
把 TtsGenerationListener 分成 fetch / preprocess / kokoro / s3 四個階段, 各記耗時, 合併成一行 log, 並算出 RTF (Real-Time Factor)。透過 kubectl logs 直接拿前後對照數據。

(b) Micrometer Timer:
在 Kokoro 呼叫處加 tts.synthesize.duration Timer (tag: voice), 在 listener 入口/出口加 tts.pipeline.duration Timer(tag: voice、outcome)。

驗證方式不需要 Prometheus, 直接 port-forward 拉 actuator endpoint:

回傳會有 count、mean、max 作為改動前後的對照基線。同樣的指標等之後上 Prometheus 時自動就能 scrape, 無痛升級。

這裡補充一下:

RTF = 生成時間 / 音檔長度

是語音生成領域的標準指標:

  • RTF = 1.0: 生成 1 秒音檔要 1 秒, 基本上是實時生成
  • RTF < 1.0: 比實時快, 越小越快
  • RTF > 1.0: 比實時慢, 使用者要等

舉例一個 3:06（186 秒）音檔，跑了 107 秒, RTF = 107 / 186 ≈ 0.575.

表示生成只用了實時的 57.5%，亦即比即時播放快 1.74 倍。

測試

階段 fetch preprocess kokoro s3 total audioSec RTF
Phase 0 baseline 88 24 496232 1273 497621 747.4 0.664

解讀一下數據:

  • audioSec = 747.4s (12:27 音檔)
  • chars = 11,975 (~1.2 萬字, 痾確實很長…)
  • kokoro = 496,232ms ≈ 8 分 16 秒
  • total = 497,621ms ≈ 8 分 17 秒
  • fetch + preprocess + s3 = 1,385ms (0.28%)
  • RTF = 0.664

結論明確:

Kokoro 占 99.7% 總時間, article-service Feign / preprocessor / Garage S3 都不是問題, RTF 0.664 完全符合「單緒 CPU」預測(單緒典型 0.5–0.7), 跟 polling ~202 次的觀察差不多 (202 × 2.0s ≈ 404s, 實際 497s, polling 是抓概數而已).

下一步則是藉由修改 Kokoro Deployment, 讓 Kokoro 用滿 CPU limit (4 vCPU).

優化 Kokoro cpu

基本上直接修改 OpenMP / ONNX / MKL / OpenBLAS thread 數.

這裡有趣的來了 !!!

階段 日期 fetch preprocess kokoro s3 total audioSec RTF
Phase 0 baseline 2026-05-05 88 24 496232 1273 497621 747.4 0.664
4 核 (cold) 2026-05-05 98 23 565452 716 566292 747.4 0.757
4 核 (warm) 2026-05-05 79 10 413927 374 414393 747.4 0.554
6 核 (warm) 2026-05-05 39 5 375268 492 375807 747.4 0.502

首先是 cold & warm, 一開始改 4 核去跑, 時間竟然還增加了快 14%, 差點吐血, 問了一下 AI, 結果是因為重啟 Kokoro 服務, 首次會將模型載入記憶體導致耗費一些時間, 在逕行第二次以及新增到 6 核心之前都做一次預熱, 就能看到 RTF 往下掉了。

But !

6 核 RTF 雖然壓到 0.502, 但改善已遞減, Kokoro CPU 推論已碰到 登峰造極 天花板, 再衝 7 核效益預估 < 5%(<20s), 但代價是其他中間件 burst 時被擠壓, 因此:

從 6:15 再往下要靠架構改變, 不是 CPU 配置。

借鏡與研究

這時我回頭去觀察了 typecast.ai, 我發現他是將各個文章依照段落切成小塊, 並且在更新時馬上就往後端打過去:

藉由小段的平行輸入, 最終整合成一整段的語音回來, 這個設計完美解決了單一段落過長的問題, 但是在整合音檔和前端上就需要額外下一點功夫。

其實我還有一招

在參考 typecast.ai 的做法後, 我內心大概知道 Beta2.0 是必要做到文章分段平行合成這個功能, 藉由 chars 切成多段, 每段抓生成時間小於 2 ~ 3 秒, 因為 ONNX 在小 batch 下對 thread 數應該沒那麼敏感, 多個小 request 的總和時間應該會 << 一個大 request.

雖然說把段落切開做語音是必要的, 但是對於目前的 effort 太大, 針對當前的問題, 我其實還有一個方法, 就是在 Beta2.0 前, 針對 Free Plan 的人想要生成語音, 先估算一下多少文字會導致等待超過 15 秒, 然後在 service 裏先實作 Strategy Pattern, 讓 Free Plan 檢查文字過多則禁止生成, 並推薦用戶建立多個 article 分散語音長度, 未來可以再把閥值拉高, 同時也能起到防止用戶胡亂輸入的情境。

遇到問題可以不用硬幹, 藉由需求更新其實能導出不同的解法

在過往的工作經驗上, 也滿常遇到類似的問題, 通常這時候會拉一個會議進行討論, 多半可以得到不錯的結果, 當然前提是我們要先提出合理的 Alternative Solutions.

最終作法

首先藉由估算與測試, 500 chars 大概會花費約 15 秒上下, 算是一個人能夠等待的最大時間, 故先更新成 Free Plan 生成語音限定在 500 chars 內.

不過 500 chars 真的很少… 就算是我自己也不太夠用, 下一次的功能更新也是勢在必行了.

其實我還有真正的最後一招

我們現在是靠著 R5 3600 的 4 core 硬上, 並且使用 kokoro-fastapi-cpu.

而為了省電, 我 Server 上安裝的是 GT1030 亮機卡, Tensor Cores 是 0, 假設我把 kokoro-fastapi 改用 gpu, 然後來一個 GPU 升級, 絕對會有感.

為此我在我的筆電上改用 ghcr.io/remsky/kokoro-fastapi-gpu:latest 做了一次測試, 先使用 cpu r5 4600h 生成一次三分鐘的 mp3, 耗時約 107 秒, 再改為使用 RTX 3050 Laptop 4GB 測試.

傻眼, 從 106.929 秒降低成 9.044 秒, 爆降快 12 倍!

看到這, 我默默打開原價屋, 看一下 RTX 3050 6GB 要多少錢…

結語

這一次算是一個 step by step 的優化, 同時也做了最適合業務現狀的妥協與規劃, 但另外一方面也是知道做這一個 Feature, 他的核心技術其實是在處理這一塊上面, 我想更新完這個 Feature 後在寫一篇文章介紹這個做法吧。

這個情境讓我想起之前在處理銀行外幣交易的時候曾遇到一個案例, 需求指名要即時獲取銀行的匯率, 但由於匯率來自 AS-400 的大型主機, 在高頻率查詢下會有顯著的效能問題, 此時研究後得知核心交易也是在 mainframe 內完成, 且最終匯率由交易當下為主(就是 mainframe 內的匯率, 非前端獲得並顯示的匯率), 因此我們就可以在 bff 內設定快取, 來避免 mainframe 效能的問題。

所以其實不用硬去優化 tcp gateway 那裏的程式, 反而靠著理解業務邏輯與溝通, 能開闢出一條新的路來。

另外真*最終絕招升級硬體, 在大多數情境下都能顯著的提升效能, 但是在真實的開發情境, 對於硬體的升級通常都有預算考量(特別是乙方公司), 不到萬不得已通常不會用, 不過使用 AI 依靠 GPU 也是在前期就該規劃到的事情, 這點倒是可以反省一下; 不過對於我自己的專案, 應該還是會買一個來試試看… 荷包要受難了.

最後感謝你閱讀到這裡, 在此開放一周的時間給我 友善的 讀者們測試, 若有問題與想法也歡迎和我分享與回報, 回報流程可以看下面的介紹, 然後麻煩不要尻爆我的 server。

另外說明一下, 我的網路是一個 4G 499 吃到飽 sim 卡, 只有可憐的 24mbps, 多數的瓶頸應該都在網路速度上
