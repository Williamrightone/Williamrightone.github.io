VoxelMD | 5. Clean Architecture 微服務單體化流程

VoxelMD 在上週完成了上線, 目前微服務 K8S 版本靠著 Github Private Runner & ArgoCD + Helm Chart, 只要 merge feature 後, pipeline 會自動打包前端成靜態檔案 scp 到 host server, 後端包為 image 推送到 Harbor;

再手動更新 helm chart image tag, commit 後 ArgoCD 就可以自動 polling 比對差異, 然後將新的 image 部署進去, 十分絲滑.

不過高敏如我有點受不了 ASUS 主機板的 RGB 光害, 以及老開著伺服器的 PTSD, 再加上前幾天 TTS 跑 10 分鐘的事件後, 讓我在緣分下入手了 Hp 的 Z2 Mini G5, W-1250(6C/12T) 32GB ram 搭載 Nvidia T1000 4GB 繪圖顯卡(大概是 GTX 1650 Laptop 版).

評估後我將 VoxelMD 從微服務 refactor 成單體架構, 也因為前期設計時有強調六角形架構, 再轉化的過程中十分順利.

得益於 T1000 896 個 CUDA Cores(但沒有 Tensor Core), 在不切語音的情境下, 生成 5 分鐘語音大概 15 秒, 3 分鐘語音大概 10 秒, 讓這個 Feature 到了我能接受的程度.

今天就來分享一下整個過程:

前言

VoxelMD 的原型是一套標準 Spring Boot 微服務:

一個 BFF、五個 biz service、Feign 跨服務呼叫、Kafka 跨服務事件、RabbitMQ 處理 async command, 每個 service 各自一個 schema.

從架構與部署上 Beta 版本都沒有太大的問題(應該吧…), 但這個產品實際還沒有任何規模壓力, 在最一開始其實也只是我為了解決 Obsidian 的 TTS plugin 不適配的問題, 於是維運成本(網路呼叫、分散式追蹤、跨服務交易、部署複雜度、環境組裝) 反而成了我要持續優化這個專案的最大阻礙.

因此我決定做一個 單體版 voxelmd-mono 並行維護。但是有兩條鐵則：

退回單體可以省下分散式系統的負擔，但不能省下分散式系統概念的設計紀律, 核心是 Clean Architecture
將來真要拆回微服務時，最好是動 port 跟 adapter 層、不是重寫業務邏輯

這篇紀錄的就是貫徹這條鐵則的過程，特別是身份識別這件事, 它接續了 VoxelMD #3 在微服務情境下對 ThreadLocal 的決定，但這個答案在單體版有了一個微妙的轉折。

1. 防止架構倒退

退回單體最容易出現的反射動作，就是把過去為了跨網路而設計的東西全部簡化掉。例如：

微服務時代的設計 退回單體時的考量 最終選擇
跨服務只透過 ID 關聯（沒有 JPA association） 同個 schema，加個 @ManyToOne 可以簡化流程 維持純 ID 關聯
Port / Adapter 把 storage / TTS / S3 抽象出來 直接 inject JpaRepository Port 是業務邊界，不是網路邊界, 應保留
跨服務交易靠 Saga / 事件編排 + Sweeper 「直接用 @Transactional 包起來」 接受, 這個交換挺合理的, 但原先在 usecase 層處理, 要審視是否下放到 Service 等級
BFF 在請求邊界提取身份 ThreadLocal 是同個 process，全層通用 拒絕。這就是這篇文章的主題

這四件事情的取捨原則其實是一樣的, 看便宜的選項會在要回到拆微服務的那天會造成多大的痛苦。前三條我盤點完都很果斷，第四條花我跟 AI 討論了一輪才收斂，所以後面會單獨拉出來細講。

不過在細講第四條之前，得先把整體呼叫鏈是怎麼聚合的講清楚, 因為身份識別其實只是這套通用結構底下的一個特例。

2. Clean Architecture：六角形架構下的呼叫鏈

先看微服務版一個跨服務操作的完整呼叫鏈：

退回單體後，BFF process 跟 Biz Service process 合一，跨網路那一段消失。

但這裡有個關鍵：不是把兩邊的層直接壓平成一條，而是：

三個重點：

  1. Service → Adapter(Repos) 完全不動。 Port 是業務邊界、不是網路邊界，所以最內圈原封不動, 這是聚合成本為零的部分。

  2. 不壓平成單層 UseCase，刻意保留雙層 UseCase。 外層 UseCase 接手原本 BFF 的編排職責（聚合、協調多個下游），內層 UseCase 是原 Biz Service 被吸收進來的業務單元。原本 Feign 那一跳，變成外層 UseCase 對內層 UseCase 的 in-process method call。第 6 節的 GetUsageUseCase 聚合 workspaceUsage / mediaUsage / recordingUsage，就是這個雙層形狀的真實樣子。

    為什麼不順手做成一層？因為壓平後就把編排(orchestrator)跟業務單元(Domain usecase)兩個職責縫銲死在一起, 保留雙層接縫才還留在原位

  3. 接縫被刻意留在兩個 UseCase 之間。 將來真要拆回微服務，要動的只是在這條接縫上重新插回 Adapter(Feign) + remote Controller，下游基本上完全不需要碰。

對照一下各層在單體化跟再拆回去微服務兩個方向的成本：

層 單體化時的處置 再拆微服務時的成本
Service → Adapter(Repos) → DB 完全不動 0
內層 UseCase（原 Biz 業務單元） 從被 Feign 叫變成被 in-process 叫 0（介面不變）
外層 UseCase（原 BFF 編排） inject 對象從 Feign client 換成下游 UseCase 換回 Feign client
原 Biz Controller + Adapter(Feign) 聚合消失 在雙 UseCase 接縫處重新插回

這就是開頭那條鐵則的結構面落實：退回單體省掉的是分散式的實作，不是分散式的邊界。 邊界（Port、UseCase 接縫）原地保留，trade-off 就被壓到只剩接縫上插不插 Feign 等決定。

實際上, 我之前參與過的台灣交通支付系統, 也是類似這樣的架構.

而身份識別這條路徑，是上面這套通用結構的一個特例, 它的接縫不在 UseCase 之間，而在 Controller 那一行。接下來整篇講的就是這個特例，先從上一篇的結論接起。

3. 上一篇的結論回顧

VoxelMD #3 的核心結論是：

ThreadLocal 屬於 framework 級別的橫切關注，不應該成為業務邊界的傳遞機制
跨網路邊界請用網路協定（HTTP header），跨函式呼叫請用方法參數

那篇的具體決定是：

  • BFF 入口：JwtAuthFilter 解 JWT，寫進 UserContextHolder。ThreadLocal 在這裡合理，因為它是同進程同請求的暫存。
  • BFF → biz service：跨網路邊界，改走 @RequestHeader("X-User-Id")，徹底拔掉 UserContextFeignInterceptor 這種從 ThreadLocal 偷塞 header 的隱式行為。
  • Biz service 內部：拔掉所有 InternalAuthFilter 跟下游 UserContextHolder。Controller / UseCase / Service 全部把 userId / Plan 做成顯式參數。

那次重構動了 30+ 個 method signature, 換來的是身份識別這條路徑上，整個系統只有一個地方接觸 ThreadLocal — 就是 BFF 的 filter.

4. 而在單體裡, BFF 是誰？

或者該說, 誰要擔任 orchestrator? (BFF 最原始的概念其實是服務於 Frontend, 例子是 Web, Mobile 各一個)

通用呼叫鏈的聚合第 2 節已經談過了，這裡只把身份相關的層標出來。微服務版的拓撲：

單體版沒有 BFF process，但 JWT 還是要在某個地方解開、放進 holder：

問題剩兩個：

  1. 誰 該負責把 holder 的值取出來，往下游 propagate？
  2. 誰 之下，全部禁止接觸 holder？

第二個問題比較好答, 跟微服務版一樣：Service / Domain / Adapter 全部禁止觸碰 holder。它們只接顯式參數。只要這條規則只要守住，未來拆服務時這幾層幾乎不會被影響。

而第一個問題的答案, 就是 UseCase 了…. 嗎?

5. 直覺下, 用 UseCase 讀 holder

我的第一直覺是 UseCase 是單體版的 BFF。

如第 2 節所述，BFF 的編排職責在單體版被外層 UseCase 接手。既然 BFF 在微服務時代還兼著處理 request-bound concerns(auth、rate limit、identity 提取)，那很自然會推論：UseCase 作為 BFF 的繼承者，也應該繼承 BFF 在請求邊界讀 ThreadLocal 這個職責, 而不是讓 Controller 多承擔這件事, Controller 應該只做純 HTTP 翻譯。

這個方案我們暫且叫它 Option B。寫起來會像這樣：

看起來很乾淨。但跑了一輪 grep 後我發現它在這個 codebase 上會撞牆。

6. 跨模組 + async 的 UseCase 撞牆

罪魁禍首是這個 usecase (又是你, 你最爛, 人家西點軍校你西點麵包)：

這正好是第 2 節講的「雙層 UseCase」的真實樣子 — GetUsageUseCase 是外層編排，workspaceUsage / mediaUsage / recordingUsage 是內層業務單元。而且讀過上一篇就知道這個 pattern 就是讓 ThreadLocal 失效的那個 supplyAsync 跨執行緒場景。USAGE_EXECUTOR 上的 worker thread 跟原本的 request thread 不是同一條，導致 holder 是空的。

那篇文章最後我決定 不用 TaskDecorator 自動傳 ThreadLocal.

這個決定還生效中 — 所以在文件 auth-context.md 還是寫著 identity does not propagate, intentionally。

問題來了：workspaceUsage.execute(userId) 這個被 async 呼叫的 UseCase，它能不能讀 holder？ 不能。它在 worker thread 上，holder 是空的。

但同時這個 GetWorkspaceUsageUseCase 也被 WorkspaceUsageController 直接呼叫（同步、在 request thread 上、holder 有值）。它有兩個 caller，分別處在 holder 有值跟 holder 無值兩種環境。

如果走 Option B（讓 UseCase 讀 holder），這個 UseCase 就沒地方擺：

  • 規則化成: top-level UseCase 讀 holder、inner UseCase 接參數? 這個 UseCase 兩種角色都是
  • 規則化成: 全部 UseCase 都讀 holder? async caller 直接壞

不對稱規則站不住。我跟 AI 討論了一輪，剩下兩條路：

(a) 我就接受不對稱：跨模組 / async-callable 的 UseCase 接參數，純 controller-only 的 UseCase 讀 holder。但同一個 UseCase 兩種角色都有的時候，這個分類就崩了。

(b) 換規則：所有 UseCase 統一接顯式 Long userId 參數，Controller 變成那個唯一讀 holder 的層。

最後選了 (b)。叫它 Option A。

7. Option A：Controller 是身份的唯一入口

具體形狀：

整個系統的 UserContextHolder.get() / .currentUserId() 呼叫，現在只出現在兩個地方：

  • JwtAuthFilter（寫入 holder） — 1 處
  • 所有 controller method 的開頭（讀取 holder） — 11 個檔案

UseCase / Service / Domain / Adapter — 零 holder 存取。這條規則用 grep 就能驗證，等於用 IDE 工具 enforce 邊界紀律。

8. 又把 holder 塞回 Controller

我前面才說把 UserContext 放在 controller 內很髒，應該要在 usecase 內呼叫 UserContextHolder，結果現在反而保留這樣的做法？

這個問題很尖銳。我承認自己前面講對於 dirty 的定義太敏感.

行為 是否 dirty
@CallerUserId Long userId 由 Spring resolver 從 holder 偷塞給 controller method 參數 是。Method signature 看不出 userId 來源，是 Spring magic 製造的隱式耦合
Service 層 / Domain 層直接 UserContextHolder.get() 是。純邏輯層依賴全域狀態，testable 性差，未來拆服務直接崩
Controller method body 裡一行 Long userId = UserContextHolder.currentUserId(); 不是。這是 hexagonal 架構裡 primary adapter 的標準工作, 把外部請求概念翻譯成內部 domain 輸入

關鍵差別還是在前兩種把骯髒藏起來假裝乾淨；第三種把骯髒精確收在唯一一條邊界線上。 後者才是 hexagonal / clean architecture 的核心精神。

回頭看上一篇文章，那篇砍掉的也不是 BFF 讀 ThreadLocal 這件事 — BFF 那一份 UserContextHolder.get() 一直保留著。砍掉的是 跨服務 Feign interceptor 偷塞 header 跟 下游 service 重新放一份 holder, 所以按照同樣的精神：holder 只能存活在請求邊界，不能擴散到請求邊界以外的地方。

另外還有一個更重要的觀點:

未來若 controller 變成 biz 服務之內，本身他的 usecase 對 UserContext 來源也會是 controller，所以當前先暫時讓服務從 controller 內丟進去，也是合理的

微服務版裡 biz service 的 controller 長這樣：

它做的事情就是從 request 取出 caller identity → 傳給 use case。跟單體版的 controller 結構完全一樣，差別只在外部請求概念是 header 還是 holder（而 holder 也是 filter 從 JWT 翻來的）。

換句話說：單體版的 Controller = 微服務版的 Biz Service Controller。它在 hexagonal 架構裡占的位置一直都是 primary adapter，不管底下是同進程還是跨網路。

9. 業界作法對照：Spring Boot 自己也是這樣寫的

決定 Option A 之後我做了一輪 cross-check，看這個 pattern 跟業界主流的 alignment 程度。結果其實非常一致：

Spring Security @AuthenticationPrincipal

Spring 官方教學裡 controller 永遠長這樣：

Controller 接 principal 物件，往下傳 user.getId() 給 service。Service 自己不碰 SecurityContextHolder。Spring 自家的 sample / Spring Boot starter / Spring Cloud 教材幾乎全是這個 pattern。

註：SecurityContextHolder 跟我自己寫的 UserContextHolder 結構幾乎一樣 — 都是 ThreadLocal，都靠 filter 寫入、finally 清掉。Spring 自己對 holder 的態度也是 inbound adapter 才碰，往下走顯式參數。

Hexagonal Architecture（Alistair Cockburn）

原文定義：

Primary adapters drive the application. They translate external request formats — HTTP, CLI, gRPC, message queues — into the application’s port-defined inputs.

翻譯外部請求格式成應用程式的 port 輸入, 字面上就是 controller 該做的事。HTTP / Header / JWT / ThreadLocal 本來就屬於外部，不該滲到 application 層以下。

Clean Architecture（Uncle Bob）

外圍的 framework/driver 層才能接觸 HTTP / SecurityContext / ThreadLocal。內層（UseCase / Domain）的依賴方向永遠朝外，介面只接受純資料 — Long userId、不是 HttpServletRequest。

BFF Pattern

Sam Newman 的 Building Microservices 跟 ThoughtWorks 的 BFF 文獻裡：

BFF 是身份的入口，下游 service 的 controller 是身份的再入口, 這也正是我前面講的 Biz Service Controller = 單體 Controller。

Domain-Driven Design（Vaughn Vernon）

Implementing Domain-Driven Design 第 14 章：

The Application Service is the boundary between the outside world and the domain model.
It receives identity and authorization information as method parameters from the inbound adapter.

DDD 對 application service 的定義就是接 inbound adapter 給的顯式 identity。

共通結構

把這些教科書放一起看會發現他們在描述同一個東西，只是用詞不同：

來源 「身份提取的唯一邊界」 「下游純參數」
Spring Security @AuthenticationPrincipal service.do(user.getId())
Hexagonal Primary adapter Application port
Clean Architecture Frameworks & Drivers 層 Use Cases 層
BFF Pattern BFF + biz service controller Biz service logic
DDD Inbound adapter Application Service
VoxelMD Mono Controller from UserContextHolder UseCase / Service

每個做法都有同樣的核心結構：
身份在邊界提取一次，往下全部顯式參數。 這就是為什麼 Option A 看起來簡單，但其實是業界最深的共識之一。

10. 拆回微服務的零成本路徑

第 2 節談過通用結構的接縫在雙 UseCase 之間。身份這條路徑是它的特例 — 接縫換到了 Controller 那一行。兩者是同一個論證的兩個投影：邊界原地保留，拆回去就只是替換邊界上「外部來源」的字面表示。

既然 Controller 在 hexagonal 架構裡的位置不會變，未來真要拆回微服務時，身份這條路徑的整個遷移成本就只剩這一行：

下游所有層: UseCase、Service、Domain、Adapter 就能完全不變，因為它們本來就只接顯式參數。如果當初讓 service 層偷讀 holder，那拆服務時這些層每一個都要重寫。

我大概想了一下各 Option 的 trade-off.

設計選擇 拆服務時的成本
Service 層讀 UserContextHolder 每個 service method 重寫，可能上百個檔案
Controller 接 @CallerUserId Long userId（annotation magic） 換成 @RequestHeader("X-User-Id")，但 magic resolver 要刪
Controller 內顯式 UserContextHolder.currentUserId() 改一行字面字串成 @RequestHeader

第三條贏的理由是：身份提取的責任已經精準收在 Controller method body 的開頭，移植時只要替換「外部來源」的字面表示，邊界位置完全沒變。

結語

這次重構動了 11 個 controller、4 個 UseCase 介面、4 個 UseCase 實作、1 個 doc rule，加上刪 4 個基礎設施檔（@CallerUserId annotation + resolver + WebMvcConfig）。比上一篇的 30+ 還少，但花的討論跟思考的時間更多, 因為每個決定都要過一遍拆服務時, 現在下的決策會導致什麼後果.

等後續 VoxelMD 的 Feature 開發到一定規模, 我再做一次把他從單體拆分到微服務的流程好了, 希望不會啪啪打自己的臉 XD.

謝謝你看到這裡, 後面的開發 VoxelMD 基本上都會走單體架構了, 近期 VoxelMD 相關的文章(若還有)都會以 Feature 為主, 下一篇文章也正在撰寫中, 是我把 Excalidraw Canvas Feature 引入的心得.
